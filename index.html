Blogs Made Easy

My latest project (Blogs Made Easy) was something I created for my portfolio.

It's a Command Line Interface (CLI) tool, that I wrote completely in Rust; it parses a template HTML file, and a list of Markdown files. By using placeholders within the template, and adding corresponding values in the Markdown, it will create a new HTML file for each Markdown.
This is ideal for me as I write my blogs in Markdown, but can ensure that the template that I have is the same for all blog entries; or if I were to update the template, that change can be reflected across the entire site.

I am currently working on a rework of the library so that I can create both a binary and a Web Assembly (WASM) build too.

My Values and Beliefs

Old School thinking

Perhaps it's due to how I started in the industry, but I was very young and had to teach myself everything. As a result, I ended up learning how to do pretty much everything I wanted on my own. At the time, I thought that if I used a third party library, then I'd either have to pay or accredit them on the site, so I avoided it.

This meant that while many developers were recommending using $("#my-element") to get an element with a particular ID, I was using the less glamourous document.getElementById("my-element").

Perhaps this was also due to the fact that I was in an industry that was still using IE6 long after they should have been...
Engrained in my brain is <!--[if lte IE 6]-->. Urgh, that's made me shudder just thinking about it! For many, many years, I had a tab open for Can I Use almost permanently. Even now, I find myself double checking if I can use features that have like 95% coverage!

Do you remember a time before Flexbox? Because I do, and I never want to take it for granted!

No Favourites

I also love to learn new things, something I find both fun, but incredibly useful.
It's quite common for developers to chose a framework like sports fans support their favourite team. Rather than looking objectively at what their application is required to do, and what tools they can use for that.

For my portfolio I decided to use Pico CSS framework and just plain JavaScript.
This is aimed at being a simple site, that doesn't need many bells and whistles, but should load content quickly.
While it's possible to have used my go to Tailwind, but (as outlined in my blog post), Pico is mostly class free, meaning that I could focus on the content of the site.
Comparing this to my todo list app, which I chose Yew and Tailwind; in this case I wanted chose frameworks that were more versatile which was ideal in that scenario.

You may have also noticed that I didn't opt for a frontend framework for the JavaScript in this site; just a simple custom components.
Typically I would reach for VueJS if I wanted to create some form of reactive component, but in this case I think it would be overkill, and hence I chose to write my own components.
The use case here was just to create a tab view element, that would contain some controls which in turn would toggle the specified page into view. Relatively easy to implement (all three components come in at under 125 lines!), and with a 95% coverage on Can I Use, it feels like something that needs to be embraced more.

Good quality documentation

As I've touched on above, I very much value good quality documentation and code. I think this is courtesy of the years spent making Laravel apps, and consulting their (perfect!) documentation.

Using Blogs Made Easy as an example, I have put the installation and usage instructions at the very top; before going on to explain, in detail, each argument with accompanying code block examples.
In my opinion, concise code block examples are the most important part of any documentation. My time spent asking and answering questions on Stack Overflow, has led me to utilising this skill somewhat regularly; and, I believe, to a good standard.

Comments, and the importance of them

Comments are another part of documentation that I believe should be done with care.
I have worked on many projects where there have been comments like "create variable" or "check if variable is 'some string'". In my opinion, these type of comments should just be removed as they offer nothing of value and only aim to explain what we already know.

Personally, I believe there should be two types of comments: How and Why.

How comments (sometimes I refer to them as "What comments") should be used to explain how a block of code works, or what it does. Normally used at the start of a particular block of code, these comments are used to help any developer understand the code without having to read the code.

Why comments are used to explain why a particular decision was taken for this block of code. I probably use these types of comments the most, like this example where I explain why I am performing a sort on the offset.
